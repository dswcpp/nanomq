# 焊接高频采集场景下 NanoMQ 服务端健壮性与入库性能问题定义及优化草案

## 1. 文档目的

本文档作为“问题定义 + 优化草案”合并版，正式冻结以下内容：

- 在焊接高频采集场景中，当前 NanoMQ 服务端虽然已经具备四类遥测数据的结构化解析与 TDengine 入库能力，但尚未证明其在持续高吞吐、长时间运行、异常连接、批量入库压力下具备足够的健壮性和可恢复性
- 当前问题的主战场应放在 NanoMQ 服务端，而不是将“高频发送导致风险”简单转嫁为客户端限速问题
- 后续优化工作需要围绕“服务端高压下仍能稳定接收、解析、排队、写库并维持连接可用性”展开，而不是停留在功能可用层面
- 在问题定义冻结的基础上，给出第一版服务端优化草案，为后续专项设计和实现排定优先级

本文档分为两部分：

- 第 1～12 章：冻结问题本身、风险边界、证据基础、优化目标和验收口径
- 第 13 章：给出第一版优化草案，作为后续实现设计输入，不等同于最终代码级方案冻结

## 2. 背景

### 2.1 业务背景

本项目承载《[焊接数据采集与边缘监测数据规范](/home/tery/project/nanomq/docs/zh_CN/焊接数据采集与边缘监测数据规范.md)》中的四类常规遥测数据：

- 温湿度
- 气体流量
- 电流
- 电压

系统目标并不是简单转发 MQTT 原始消息，而是：

- 按 Topic 和 Payload 结构校验消息合法性
- 将 `data.fields + data.points` 展开为点级数据
- 将结构化结果按业务表模型写入 TDengine

这意味着 NanoMQ 在本场景中承担的不只是 Broker 角色，还承担了：

- 协议接入层
- 规则匹配层
- 业务解析层
- 入库缓冲层
- 数据库写入层

### 2.2 当前功能背景

当前仓库已经引入焊接专用解析链路，设计文档见 [taos-current-points-design.md](/home/tery/project/nanomq/docs/zh_CN/rule/taos-current-points-design.md)。

当前链路的大致结构为：

1. Broker 收到 MQTT PUBLISH
2. 规则引擎匹配 `weld/.../telemetry/...`
3. 若规则配置 `parser = weld_telemetry`，则进入焊接专用解析逻辑
4. `weld_telemetry` 对 Topic 和 Payload 做结构化校验
5. 将一条消息中的多个 `points[]` 展开为多行 `weld_taos_row`
6. 通过 `weld_taos_sink` 异步批量写入 TDengine

该链路在“功能正确性”层面已经基本成立。

### 2.3 当前问题不是“能不能写进去”，而是“高压下能不能一直写进去”

截至 2026-03-30 的现场排查，功能层已经验证：

- HOCON 规则能加载四条焊接 TAOS 规则
- `parser = weld_telemetry` 已经生效
- 四类超表模型已可按需创建
- 合法的电流消息已成功写入 `mqtt_rule.weld_current_point_*`

因此，当前最重要的问题已经从“功能缺失”转变为：

- 在高频持续上送、连接波动、异常断开、大消息、规则并发命中、数据库写入压力存在时，NanoMQ 是否仍能保持可用、稳定、可恢复

## 3. 为什么当前问题应定义为服务端问题

### 3.1 客户端不是本问题的主责任边界

在焊接边缘场景中，客户端的基本职责应是：

- 以长连接方式建立 MQTT 会话
- 按协议正常发送消息
- 尽量规范断开连接
- 按数据规范组织 Topic 和 Payload

客户端不应承担以下职责：

- 主动推断 Broker 当前写库能力
- 为掩盖服务端瓶颈而人为降采样
- 为服务端没有背压机制而自行发明复杂限速逻辑
- 通过“少发一点”“慢发一点”替代服务端可靠性建设

### 3.2 服务端是系统可靠性的收口点

NanoMQ 在本场景中是全链路的收口点：

- 所有客户端都先到 NanoMQ
- 所有业务校验都在 NanoMQ
- 所有结构化展开都在 NanoMQ
- 所有 TDengine 写入都经由 NanoMQ

因此，只要 NanoMQ 在以下任意一段存在短板：

- 连接生命周期管理
- MQTT 接收与 PUBLISH 调度
- 规则匹配
- 业务解析
- sink 队列
- 批量写库

最终表现都会是：

- 客户端“感觉连不上”
- 消息“看上去发送了但没入库”
- 服务端“端口在监听但服务体验退化”

### 3.3 高频数据不等于客户端有问题

高频采集是业务需求，不是异常输入。

如果系统目标包含：

- 焊接电流高频采样
- 连续遥测上送
- 长时间在线运行

那么服务端就必须把以下情况视为正常工况，而不是“异常使用”：

- 每秒多条消息
- 单条消息内多个点
- 多传感器并发上送
- 长时间不断流
- 网络抖动后的自动重连

因此，当前问题应被树立为：

> NanoMQ 服务端在焊接高频数据采集场景下的健壮性与高压入库能力不足，必须通过服务端架构和运行时机制优化来解决。

## 4. 当前现场证据

以下证据均来自 2026-03-30 的实际排查。

### 4.1 功能路径曾存在状态机缺陷

在 [broker.c](/home/tery/project/nanomq/nanomq/apps/broker.c) 中，Broker 在 `WAIT` 状态下对 publish 消息的判定逻辑存在缺陷：

- 业务消息进入 `WAIT`
- 但没有正确进入规则引擎处理路径
- 导致消息被提前释放
- 结果是“收到了 MQTT 消息，但没有执行规则匹配和写库”

这说明当前状态机路径在功能层已出现过关键缺陷。

### 4.2 旧二进制与新静态库共存导致运行态混淆

排查中发现：

- 测试构建目录 `build/` 因 `NANOMQ_TESTS=ON` 实际只更新静态库
- 运行命令 `./build/nanomq/nanomq` 启动的是历史旧二进制
- 旧二进制不包含最新焊接入库实现

这说明当前工程构建和运行方式本身也会放大运维风险。

### 4.3 Broker 曾出现大量半关闭连接

排查中观察到大量连接进入：

- `CLOSE-WAIT`
- `FIN-WAIT-2`

这意味着：

- 对端已经关闭连接
- Broker 侧未及时完成 socket 关闭或资源回收

半关闭连接一旦积累，会导致：

- 连接质量变差
- 端口虽然监听，但新连接体验恶化
- 运行一段时间后用户感觉“服务断了”

### 4.4 Broker 曾出现明显接收缓冲积压

现场一度观测到某条连接出现约 `1.8MB` 的 `Recv-Q`。

这说明：

- 数据已经到达内核 socket 接收缓冲区
- 但应用层没有及时读走

这类现象表明在某些负载组合下，Broker 已经出现“接收速度落后于输入速度”的情况。

### 4.5 HTTP 监听存在但应用不可用的现象

现场曾出现如下状态：

- `1883` 和 `8083` 端口处于监听状态
- 但本机对 `8083` 的 HTTP 请求超时无响应

这意味着：

- “端口在监听”不能等价于“服务处于健康状态”
- 当前系统缺少更可信的健康判定机制

### 4.6 后台启动方式影响服务稳定性

排查中发现：

- 以前台 TTY 方式启动时，服务可正常存活
- 使用 `nohup` 方式时，进程在初始化完成后很快退出
- 改用 `setsid` 脱离终端后，服务可稳定常驻

这表明当前运行模型对启动方式敏感，存在守护化兼容性问题。

### 4.7 入库链路对非法时间戳严格失败

当 `points[].ts_ms` 使用相对时间值 `1000/1001` 时，TDengine 返回：

- `Timestamp data out of range`

这本身符合设计文档要求，但也说明：

- 解析和入库链路目前没有更友好的异常分类与观测能力
- 一旦高频场景中混入非法消息，定位成本较高

### 4.8 批量写失败无重试、无计数（源码实证）

`weld_taos_sink.cpp` 后台 worker 线程的批量写入逻辑中：

```cpp
if (sql.empty() ||
    tdengine_exec(g_state.url, g_state.auth_header, sql) != 0) {
    log_error(
        \"weld_taos_sink: batch insert failed (%zu rows)\",
        batch.size());
}
batch.clear();
```

无论写入成功或失败，`batch.clear()` 均被调用。写失败的数据被静默丢弃，无任何重试、失败计数器或告警机制。

### 4.9 队列无上限（源码实证）

`WeldState` 结构体中：

```cpp
std::deque<WeldQueuedRow> queue;
```

该 `std::deque` 无任何容量约束。每条入队操作（`weld_taos_sink_enqueue_row_with_config`）都在持锁后直接 `push_back`，无背压判断。若 TDengine 写入速度持续落后于入队速度，内存将无限增长直至 OOM。

### 4.10 Unknown stable 仍存在整批失败分支（源码实证）

`build_batch_insert_sql` 中：

```cpp
case StableKind::Unknown:
default:
    return std::string();
```

当前正常链路下，`validate_config_and_row` 已在入队前拒绝未知 `stable`，因此这不是现阶段主热路径风险；但从防御性设计看，一旦未来有绕过前置校验的改动或错误数据直接进入批处理，该分支仍会让整批 SQL 构建失败，且日志中仅有 `batch insert failed`，无法区分是 `stable` 错误还是数据库写入失败。

## 5. 当前链路的结构性压力点

### 5.1 接收层压力点

Broker 必须同时处理：

- TCP Accept
- MQTT CONNECT / DISCONNECT
- PUBLISH 解码
- SUB 转发
- 系统主题通知

若接收层与后续解析/入库链路耦合过深，会出现：

- 收消息慢
- 断连清理慢
- 状态机阻塞放大

### 5.2 规则层压力点

当前每条业务消息仍要经过规则引擎匹配。

对于焊接场景：

- Topic 命中是高频常态
- 电流/电压共享 `power` 主题前缀
- 规则层和 parser 层都参与路由判断

因此规则层的每次计算成本都会被高频放大。

### 5.3 解析层压力点

[weld_telemetry.c](/home/tery/project/nanomq/nanomq/weld_telemetry.c) 当前会对每条消息执行：

- Topic 分段解析（`weld_parse_topic`：`strtok_r` 分割9段，校验 `weld` 前缀和 `telemetry` 位置）
- JSON 解析（`cJSON_ParseWithLength`）
- 公共字段校验（`gateway_id`/`device_id` 与 topic 一致性校验，`signal_type` 合法性校验）
- `fields[]` 线性索引建立（`weld_find_field_index`：对每个字段名做全量 O(n) 扫描）
- `points[]` 展开（每个 point 分配独立 `weld_taos_row`）
- 时间戳去重检查（仅针对 `ts_ms` 模式，`ts_us` 模式不检查重复；使用 O(n²) 嵌套循环）
- 多行结构体构造（`nng_zalloc` 分配 `point_count * sizeof(weld_taos_row)`）

**已知行为约束（源码层面）：**

- `recv_ts_us` 在消息入口处仅取一次（`weld_now_epoch_us()`），同一条消息的所有 points 共享相同的 `recv_ts_us`，无法区分 points 之间的接收时序差
- `ts_ms` 以 `ts_ms * 1000LL` 转换为微秒精度，不会从消息结构中提升实际精度
- 若同一条消息中出现重复 `ts_ms`，则整条消息的全部 points 被丢弃（`goto done`），不会部分写入
- `ts_us` 模式不执行重复检查（即允许同消息内存在相同微秒时间戳）

这条路径在大消息（多 points）、高频消息场景下 CPU 成本并不低，且内存分配点集中，失败时无降级。

### 5.4 sink 层压力点

[weld_taos_sink.cpp](/home/tery/project/nanomq/nanomq/weld_taos_sink.cpp) 当前实现模型为：

- **全局单例状态**（`WeldState g_state`）：单一全局对象，包含 URL、auth、DB、超表集合、队列、worker 线程
- **无界队列**：`std::deque<WeldQueuedRow>`，无任何容量上限，高压下内存无限增长
- **单后台 worker 线程**：串行写库，无并发写入能力
- **批量参数硬编码**：`WELD_BATCH_SIZE=100`、`WELD_FLUSH_MS=100ms`，不可运行时配置
- **写失败静默丢弃**：批量 INSERT 失败仅 `log_error`，立即 `batch.clear()`，无重试、无失败计数、无告警
- **Unknown stable 缺少精确分类**：当前正常链路会在入队前拒绝未知超表名，但 `build_batch_insert_sql` 仍保留返回空字符串的失败分支；若未来有数据绕过前置校验，整批会失败，且日志无法区分是超表名错误还是数据库写入失败
- **不支持连接目标热切换**：若 sink 已启动，再次入队时若 `host/port/db` 不同，直接返回 `-1` 拒绝，不支持重新配置
- **停止时无保障 flush**：`stop_all` 在 worker 退出后一次性写入剩余队列，该写入无重试；若此时 TDengine 不可达则剩余数据静默丢失

该模型实现简单，但缺乏生产级健壮性所需的背压、重试、可观测性机制。

### 5.5 数据库层压力点

TDengine 写入链路涉及以下步骤，每个步骤均有失败点：

- **建库/建超表**（`ensure_database_ready`）：最多重试 20 次，每次间隔 100ms（最长等待 2 秒），超时后返回 `-1` 并清空状态，后续入队全部被拒绝
- **子表名推导**（`make_sub_table_name`）：`{stable}_{gateway_id}_{device_id}`，总长 >192 字节时截断前 175 字节并追加 `_` + 原始完整名称哈希的前 16 个十六进制字符，截断不可逆
- **批量 SQL 拼接**（`build_batch_insert_sql`）：纯内存字符串拼接，超大批次时 SQL 长度不受限
- **REST 请求发送**（`tdengine_exec`）：同步阻塞调用，若 TDengine 超时，worker 线程阻塞期间队列持续增长

若 REST 超时、数据库抖动或 SQL 过大，当前系统仅记录错误日志，对上游几乎没有反馈、背压或治理。

## 6. 问题定义

### 6.1 核心问题

当前 NanoMQ 在焊接高频采集场景下存在如下核心问题：

> 在持续高吞吐、客户端波动、复杂规则解析和 TDengine 写入压力并存的情况下，服务端尚未建立完备的连接管理、背压控制、队列治理、失败恢复、运行观测和容量边界，因此无法证明其具备长期稳定接入并成功入库的生产级健壮性。

### 6.2 问题不应被弱化为“客户端发太快”

以下表述不应作为问题定义：

- 客户端不该发这么快
- 采集频率太高了
- 单条消息太大了所以 Broker 顶不住

上述说法最多只能视为“输入压力条件”，不能代替系统问题本身。

正确的问题定义应是：

- 服务端是否具备承接该压力条件的能力
- 若暂不具备，边界在哪里
- 边界被触达时，服务端以什么方式退化
- 是否存在明确的背压、丢弃、重试或隔离策略

### 6.3 问题的本质

问题的本质不是单一 bug，而是系统性能力缺口，主要体现在：

- 连接生命周期治理不足
- 运行时可观测性不足（无队列深度、无写入成功/失败计数、无延迟统计）
- 高压输入下缺少显式背压（`std::deque` 无界，入队无阻塞）
- 解析与入库的异步边界仍不够清晰（写失败静默丢弃，上游无感知）
- 单 worker 串行写库，批量写失败无重试
- 构建、启动、守护方式不够稳健

## 7. 本次问题定义的范围

### 7.1 纳入范围

本次服务端问题定义纳入以下范围：

- MQTT 连接建立、断开、异常断连、半关闭连接治理
- Broker 对高频 PUBLISH 的接收能力
- 规则引擎在焊接场景下的执行路径稳定性
- `weld_telemetry` 解析链路的高压表现
- `weld_taos_sink` 的队列、批量写入、失败处理
- TDengine 入库成功率和时延
- 后台运行、守护化、日志落地和健康检查

### 7.2 不纳入范围

本次问题定义暂不纳入以下内容：

- 波形消息专用通道设计
- 非 TDengine 数据库支持
- 云端业务系统消费逻辑
- UI 端美化和交互优化
- 传感器硬件精度与现场采样真实性

## 8. 服务端优化必须回答的关键问题

后续优化工作必须回答以下问题。

### 8.1 连接层

- Broker 是否会长期积累 `CLOSE-WAIT`
- 对端断连后多久完成 fd 回收
- 高并发短连场景下 Broker 是否仍可服务长连接客户端
- 是否需要为焊接接入场景单独调优 keepalive、连接超时和 socket 行为

### 8.2 接收与调度层

- Broker 能稳定承受的每秒消息数是多少
- 在订阅转发和规则处理并存时，接收线程是否会被拖慢
- 大消息是否会显著影响正常小消息的接收延迟

### 8.3 规则与解析层

- 每条消息的规则匹配和解析耗时是多少
- 是否存在需要提前分流的路径
- 电流、电压等高频消息是否应走更轻量路径

### 8.4 sink 层

- 队列深度多少时属于健康（当前无任何计量）
- 队列满了怎么办（当前无上限，实际上不会触发满，而是 OOM）
- 写库失败是否重试（当前：否，静默丢弃）
- 重试策略是否会反过来拖垮接收层
- 是否需要多 worker 或按超表分 shard（当前：单线程，四类超表共用一个写入器）
- `WELD_BATCH_SIZE=100` / `WELD_FLUSH_MS=100ms` 是否需要可配置化（当前硬编码）

### 8.5 数据库层

- TDengine 单批写入最佳批大小是多少（当前硬编码 100，未验证）
- 单批过大时的失败模式是什么（当前无 SQL 长度上限约束）
- REST 写入耗时分布如何（`tdengine_exec` 为同步阻塞调用，超时时 worker 阻塞，队列继续增长）
- 子表创建是否应在热路径之外完成（当前 `ensure_database_ready` 在入队首次触发时执行，最多重试 20 次×100ms）
- `stop_all` 时剩余队列的最后一次性写入若失败是否可接受（当前：无重试，数据丢失）

### 8.6 运维层

- 如何判定服务“真健康”而不仅是“端口监听”
- 如何让 systemd / nohup / setsid 等不同启动方式都稳定
- 如何采集并暴露关键运行指标

## 9. 后续优化的目标

### 9.1 总目标

建立一条对焊接高频采集场景友好的服务端链路，使 NanoMQ 满足以下要求：

- 长时间连续运行不出现连接堆积
- 高并发消息接入下仍可保持连接可用
- 解析和写库跟不上时能显式退化而不是无声失稳
- TDengine 入库成功率、时延和失败原因均可观测
- 服务启动、重启、后台运行行为一致且可预期

### 9.2 可靠性目标

- 连接异常时，Broker 不因少量坏连接进入整体退化
- 客户端断开后资源回收及时
- 后台启动后服务可稳定常驻
- 规则引擎和焊接解析路径不会因状态机问题被绕过

### 9.3 性能目标

- 在目标采样压力下不出现持续增长的 `Recv-Q`
- 写库线程的平均耗时和尾延迟可测量
- 队列长度在健康区间内波动，而不是无限膨胀

### 9.4 可运维目标

- 可以通过日志和指标快速判断：
  - 是否收到消息
  - 是否进入解析
  - 是否进入 sink 队列
  - 是否成功写库
  - 是否因背压被丢弃或延迟

## 10. 建议的优化优先级

后续服务端优化建议按以下顺序推进。

### 10.1 第一优先级：连接与状态机稳定性

- 修复所有已知的 publish 状态机路径问题
- 查清并治理 `CLOSE-WAIT` 滞留
- 统一后台启动与守护方式

### 10.2 第二优先级：sink 队列治理与观测

- 增加队列深度统计（当前 `std::deque` 无计量）
- 增加批量写入耗时统计
- 增加失败计数和失败原因分类（当前所有失败均仅 `log_error`）
- 增加健康日志
- 为 `std::deque` 增加容量上限，超限时执行明确的降级策略（丢弃/阻塞/告警）

### 10.3 第三优先级：高压背压策略

- 明确队列上限（具体行数或内存字节数）
- 明确超限时策略（丢头/丢尾/拒绝入队/阻塞调用方）
- 明确大批量消息的处理策略
- 考虑批量写失败的有限重试策略（避免无限重试拖垮 worker）

### 10.4 第四优先级：压测与容量边界

- 构建标准压测输入
- 得出每秒消息数、每条点数、每条 payload 大小三维边界
- 形成容量文档

## 11. 验收标准

本问题定义结束后，后续优化至少应满足以下验收条件。

### 11.1 功能验收

- 四类遥测消息在规范输入下可持续写入 TDengine
- 后台启动方式稳定
- 重启后可恢复服务

### 11.2 稳定性验收

- 长时间运行后无明显 `CLOSE-WAIT` 堆积
- 无“端口监听但应用无响应”的状态
- 异常客户端不会拖垮整体接入能力

### 11.3 性能验收

- 在目标吞吐下无长期增长的接收积压
- 批量写库性能可测量、可调优
- 高峰期失败行为可解释、可复现

### 11.4 可观测性验收

- 可以明确回答任一条消息在哪个阶段失败

## 12. 当前结论

截至 2026-03-30，结论如下：

- 焊接四类遥测的结构化解析与 TDengine 入库功能已经具备基本可用性
- 当前主要矛盾已经从“功能缺失”转变为“服务端健壮性不足”
- 客户端是否高频发送不是本问题的主责定义
- 需要正式启动 NanoMQ 服务端高压可靠性优化工作
- 源码审查已确认三个高危缺口：`std::deque` 无界增长（OOM 风险）、批量写失败静默丢弃（数据丢失无感知）、单 worker 串行写库（无并发能力且成为瓶颈）

用一句话概括当前问题：

> NanoMQ 现在已经能把焊接遥测数据写进库，但还没有被证明能在高频、长时、异常波动的真实现场里持续稳定地把数据写进库；源码层面的三个高危缺口（无界队列、写失败静默丢弃、单线程写库）是当前最直接的风险来源。

## 13. 数据处理优化方案

本章节基于源码审查结果，给出各层次的具体优化草案及实施优先级。**本章节为候选优化方向，作为后续实现设计输入，尚未实施，也不等同于最终代码方案冻结。**

### 13.1 方案 A：sink 队列治理（P0）

**问题**：`std::deque<WeldQueuedRow>` 无容量上限，`weld_taos_sink.cpp` 当前入队路径会持续追加，高压持续写入时内存无限增长直至 OOM。

**方案**：在 5-worker 分 shard 架构下，每个 shard 队列独立设置上限，超限时拒绝入队并记录该 shard 的丢弃计数。

**超限策略**：采用「丢弃最新」——拒绝本次入队，返回 `-1`，接收层感知到失败但不阻塞。

| 策略 | 优点 | 缺点 |
|------|------|------|
| 丢弃最新（推荐） | 不阻塞接收层，实现简单 | 最新数据丢失 |
| 丢弃最旧 | 保留最新数据 | 破坏 FIFO，需主动移除队头 |
| 阻塞调用方 | 零丢失 | 级联阻塞接收层，风险更高 |

**各 shard 队列上限参考值**（初始建议，后续根据可观测性数据调整）：

| shard | 上限（行） | 说明 |
|-------|-----------|------|
| current | 5000 | 高频，2 worker 并发消费 |
| voltage | 5000 | 高频，2 worker 并发消费 |
| env+flow | 2000 | 低频，1 worker 消费 |

总上限约 12000 行，各 shard 独立计量，不共用全局常量。

**涉及文件**：`nanomq/weld_taos_sink.cpp`

---

### 13.2 方案 B：批量写失败有限重试（P0）

**问题**：worker 线程批量写入失败后立即 `batch.clear()`，数据静默丢弃，无重试，无失败计数。

**方案**：在 worker 循环内，写入失败后最多重试 2 次，仍失败才丢弃并记录失败计数。

**重试间隔**：采用固定基准 + 随机抖动（jitter），例如 `50ms + rand(0..20ms)`。5-worker 架构下多个 worker 可能同时触发重试，若所有 worker 以相同间隔同步重试，会向已过载的 TDengine 形成并发写入峰值，加剧负载；引入 jitter 可将重试分散在时间窗口内。

**约束**：重试次数不超过 3 次。重试期间 worker 阻塞，对应 shard 队列继续增长，需与方案 A 的队列上限配合使用，避免重试阻塞期间队列溢出触发丢弃。

**`stop_all` 场景**：`stop_all` 对各 shard 的最后一次性 flush 同样应引入有限重试（建议 1 次），否则服务正常退出时仍可能丢失数据。

**涉及文件**：`nanomq/weld_taos_sink.cpp`

---

### 13.3 方案 C：可观测性埋点（P1）

**问题**：当前无任何运行时统计，无法判断队列健康状态、写入成功率和丢弃趋势。

**方案**：每个 shard 维护独立的统计结构（`std::atomic<uint64_t>`），涵盖：

| 字段 | 含义 |
|------|------|
| `enqueue_count` | 累计入队行数 |
| `drop_count` | 因队列满丢弃行数 |
| `batch_ok_count` | 批次写入成功次数 |
| `batch_fail_count` | 批次写入失败次数（含重试后仍失败） |
| `row_ok_count` | 成功写入 TDengine 的行数 |

每个 shard 的 worker 每 30 秒（或每 300 批）独立输出统计摘要，格式：

```
[weld_taos_sink/current]  queue=120 enqueue=5000 drop=0 batch_ok=50 batch_fail=0 row_ok=4980
[weld_taos_sink/voltage]  queue=80  enqueue=4800 drop=0 batch_ok=48 batch_fail=0 row_ok=4750
[weld_taos_sink/env_flow] queue=5   enqueue=200  drop=0 batch_ok=2  batch_fail=0 row_ok=200
```

各 shard 独立输出，避免混合统计掩盖单类数据的积压或丢弃问题。全局混合统计在多 shard 架构下价值有限，不建议保留。

**涉及文件**：`nanomq/weld_taos_sink.cpp`、`nanomq/weld_taos_sink.hpp`（可选暴露各 shard 统计查询接口）

---

### 13.4 方案 D：解析层性能优化（P2）

#### D1：fields 索引建立 O(n) 优化

**问题**：`weld_find_field_index` 对 `fields[]` 做线性扫描，六个字段调用六次，合计 6×O(n)。

**方案**：一次遍历 `fields[]` 建立名称→下标映射，批量取出所有字段下标，总开销降为 1×O(n)。

**影响**：消除冗余遍历，对 `fields` 数组较长时收益更明显。

**涉及文件**：`nanomq/weld_telemetry.c`

#### D2：ts_ms 去重 O(n²) 优化

**问题**：时间戳去重使用嵌套循环，最坏复杂度 O(n²)，对大批次 points 影响显著。

**方案一（协议保证升序时）**：若数据规范要求 `points[]` 按时间升序排列，只需检查相邻元素是否相等，O(n)，零额外开销。

**方案二（通用）**：收集完所有时间戳后排序，检查相邻重复，O(n log n)。

**建议**：确认数据规范是否保证 `points[]` 升序后再选择方案；若规范未冻结升序约束，则默认采用通用方案，避免把性能优化建立在未明确的协议假设上。

**涉及文件**：`nanomq/weld_telemetry.c`

---

### 13.5 方案 E：并发写库——5-worker 分 shard 架构（P3）

**问题**：单 worker 线程串行处理四类超表的写入，电流/电压高频数据会占满批次，阻塞温湿度/流量数据的写入通道；单 HTTP 连接同步阻塞时整条写入链路停摆。

**架构**：按数据频率特征将四类超表分为 3 个 shard，current 和 voltage 各配 2 个 worker，env+flow 共用 1 个 worker，共 5 个 worker 线程：

```
current队列  → worker_curr_1 ─┐
              → worker_curr_2 ─┤
voltage队列  → worker_volt_1 ─┤→ TDengine (REST)
              → worker_volt_2 ─┤
env+flow队列 → worker_env_flow┘
```

**批次分配模型**：同一 shard 的多个 worker 竞争消费同一队列。每个 worker 独立持锁取一个批次（最多 `WELD_BATCH_SIZE` 行）后释放锁，随即发起 HTTP 请求（阻塞）。两个 worker 的 HTTP 请求并发提交到 TDengine，互不等待。批次分配是「各自取各自的批次」，不是「两个 worker 共同处理同一批次」。

**env+flow 混合批次**：env 和 flow 两类数据共用一个队列，同一批次中可能包含 `weld_env_point` 和 `weld_flow_point` 两种超表的行。`build_batch_insert_sql` 按行逐一 switch `StableKind`，支持批次内混合超表，无需修改 SQL 构建逻辑。

**`stop_all` 多 shard 停止顺序**：
1. 向所有 3 个 shard 发送停止信号（设置 `running=false`，notify_all）
2. join 所有 5 个 worker 线程（等待各自完成当前批次）
3. 各 shard 独立 flush 剩余队列（带有限重试，见方案 B）
4. 清空各 shard 状态

不得仅 join 部分 worker 或遗漏某个 shard 的 flush，否则会静默丢失数据。

**优点**：
- 高频与低频数据写入通道完全隔离
- current/voltage 各 2 个并发 HTTP 连接，写入吞吐翻倍
- 各 shard 独立统计，可精确定位哪类数据出现积压

**缺点**：
- 资源增加（3 队列、5 线程、3 套统计）
- 全局单例 `WeldState` 需重构为 shard 数组，接口改动较大
- 需实测 TDengine REST 接口在同一超表多并发连接下的行为（通常支持，但需验证）

**依赖**：E 的实施依赖 C（可观测性）先就位；多队列上线后若无 per-shard 统计，无法诊断各 shard 的健康状态。

**涉及文件**：`nanomq/weld_taos_sink.cpp`、`nanomq/weld_taos_sink.hpp`

### 13.6 实施优先级

| 优先级 | 方案 | 改动量 | 预期收益 |
|--------|------|--------|----------|
| P0 | B：写失败有限重试 | 小 | 消除瞬时 TDengine 抖动导致的数据丢失 |
| P0 | A：队列上限+丢弃 | 小 | 消除 OOM 风险，超限行为可预期 |
| P1 | C：可观测性埋点 | 小 | 支撑后续所有调优决策；E 实施的前置条件 |
| P2 | D1：fields 索引优化 | 小 | 解析热路径 CPU 降低（6×O(n)→1×O(n)） |

| P3 | E：5-worker 分 shard | 大（需 C 先就位） | 高频/低频写入通道隔离，current/voltage 并发写入吞吐翻倍 |

### 13.7 验收标准

- **方案 A**：高压测试后日志中出现 `drop_count > 0` 而非 OOM 崩溃
- **方案 B**：模拟 TDengine 短暂不可达后恢复，`batch_fail_count` 不持续增长，数据在重试内恢复写入
- **方案 C**：服务运行期间每 30 秒日志中可见统计摘要，所有字段数值符合预期趋势
- **方案 D**：使用 perf 或 callgrind 验证解析热路径 CPU 占用下降
- **方案 E**：current 和 voltage 各自高频写入时延不受对方影响；env/flow 写入延迟不受 current/voltage 阻塞；current 队列 2 个 worker 并发写入吞吐量较单 worker 有可测量提升；TDengine REST 并发连接行为符合预期
